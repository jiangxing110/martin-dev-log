# Vibe Kanban 使用指南

## 简介

Vibe Kanban 是一个面向开发者的看板工具，用来把任务、代码工作区和 AI 编码代理串起来。它的核心思路很简单：先把工作拆成 issue，再从 issue 创建 workspace，让编码代理在独立工作区里干活，最后你在界面里审查改动并合并。

如果你平时会同时跑多个 AI 编码任务，这个工具的价值会更明显。它帮你把“任务管理”和“代码执行”分开，避免任务散在聊天记录里。

---

## 先准备什么

在开始之前，建议先确认这几样东西：

- 本机有 Node.js 环境
- 你准备使用的 coding agent 已经安装好
- 项目仓库已经本地可访问
- 如果你要用云端功能，准备好 GitHub 或 Google 登录

---

## 启动方式

最基础的启动命令是：

```bash
npx vibe-kanban
```

启动后，Vibe Kanban 会在本地打开界面。第一次进入时，它通常会让你先确认几个偏好设置，比如：

- 默认 coding agent
- 默认 IDE
- 通知方式

这些设置后面都可以再改，不用一开始就纠结到位。

---

## 登录与跳过登录

你可以直接用 GitHub 或 Google 登录。

如果你暂时不想登录，也可以选择继续使用未登录模式。这样仍然能创建 workspace，但有些云端能力会不可用，例如：

- kanban board
- issues
- 团队协作功能

如果你的目标只是先体验本地工作流，跳过登录也够用。

---

## 核心概念

### Issue

Issue 是任务本身，可以理解为“要做什么”。它至少包含：

- 标题
- 描述

还可以补充：

- 优先级
- 标签
- 父子关系

适合把 bug、feature、优化、一次性小任务都放进来。

### Workspace

Workspace 是“实际做事的地方”。它会把你选定的仓库拉成 git worktree，并启动 coding agent，让代理直接在隔离环境里执行任务。

一个 issue 可以对应多个 workspace，这对较大的功能很有用，因为你可以并行跑多个代理。

Workspace 也可以不绑定 issue，适合临时问答、探索代码库、快速试验。

---

## 基本使用流程

### 1. 创建 issue

先在看板里点新建 issue，填上标题和描述。

建议写法尽量具体一点：

- 目标是什么
- 现在的问题是什么
- 期望结果是什么
- 有没有限制条件

这样后面交给 agent 时，执行质量会更稳。

### 2. 选择 issue 并创建 workspace

选中刚才建好的 issue，然后进入 workspace 创建流程。

你需要指定：

- 要工作的仓库
- 仓库所基于的分支
- coding agent 配置

创建后，Vibe Kanban 会自动：

- 创建 git worktree
- 启动代理
- 把任务 prompt 发给代理开始执行

### 3. 查看执行过程

创建成功后，你可以在 workspace 中看到代理的执行日志和上下文。

如果任务比较复杂，可以继续补充信息，或者把 issue 再拆成更小的子任务。

### 4. 审查改动

当代理产出代码后，你可以切到 workspace 视图，查看：

- 代码差异
- 文件改动
- 运行中的预览页面

如果项目有前端页面，建议配合内置浏览器或者本地预览去看效果，而不是只看 diff。

### 5. 合并结果

确认没问题后，可以：

- 直接开 GitHub pull request
- 或者在本地把 workspace 分支 merge 回去

## 适合怎么用

### 适合的场景

- 你要同时跑多个 AI 编码任务
- 你希望任务、工作区、审查流程分开管理
- 你想把一个大功能拆成多个 issue 并行推进
- 你需要更清楚地看代理到底在做什么

### 不太适合的场景

- 只是临时跑一个很小的命令
- 你不需要任务管理，只想直接和一个 agent 聊代码
- 你目前没有可用的仓库和分支

---

## 实操建议

### 写 issue 的时候

尽量把 issue 写成“可执行指令”，不要只写一句抽象目标。

比如不要只写：

- “优化这个页面”

更好的写法是：

- “把列表页的筛选条件改成三列布局，并保持移动端可折叠”
- “为订单详情页补一个空状态和错误状态”

### 开 workspace 的时候

尽量选对分支和仓库，避免代理在错误基线上的工作白费。

如果任务跨度比较大，建议给同一个 issue 开多个 workspace，分别让不同代理负责不同部分。

### 审查的时候

不要只看代码有没有改，还要看：

- 逻辑是不是按 issue 目标实现了
- 有没有多改不必要的内容
- 预览页面有没有真实跑通
- 是否需要补测试或补说明

---

## 常见注意点

### 1. 未登录模式功能有限

如果你不登录，能跑本地工作流，但团队协作和看板能力会受限。

### 2. Workspace 会创建 worktree

这意味着每个 workspace 都相对独立，不会直接污染你当前主工作区，适合并行任务。

### 3. 一个 issue 可以对应多个 workspace

这不是重复劳动，而是一个很适合大任务的设计。你可以让多个代理并行试不同方向，最后保留最好的结果。

### 4. 代码和浏览器预览最好一起看

只看 diff 容易漏掉交互、样式和运行时问题。能预览就尽量预览。

---

## 一句话上手版

如果你只想先跑通一次完整流程，就按这个顺序来：

1. `npx vibe-kanban`
2. 进入界面并完成偏好设置
3. 登录，或者先跳过登录
4. 新建一个 issue
5. 从 issue 创建 workspace
6. 让代理开始执行
7. 在 workspace 里审查改动
8. 合并结果

---

## 建议的下一步

如果你接下来是想真的在本地体验一遍，最值得做的是：

- 先准备一个小而具体的 issue
- 选一个已有仓库
- 用一个简单任务跑通完整闭环

这样你很快就能知道这个工具是不是适合你的工作流。
