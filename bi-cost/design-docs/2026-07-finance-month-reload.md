# 金融渠道成本按月自动重洗方案

## 摘要

当前 `flink/total_cost/finance` 下的金融渠道成本脚本大多把 `source_month` / `next_month` 固定写死，导致后补录历史月份的 `ods_bi_month_tag` 数据时，无法自然感知并重算受影响月份。
同时，删除类变更也需要兼容：
- 软删除：只清理对应月份的目标历史数据，不再插入新数据。
- 硬删除：通过 CDC / 快照 diff 找回被删记录对应的月份后，同样只清理对应月份的目标历史数据，不再插入新数据。

本方案将月参数改为从 `ods_bi_month_tag.statistics_time` 自动归月推导，按 `source_month` 维度生成完整月窗口，并让各业务拆分脚本在一次执行内覆盖所有受影响月份，从而支持补录历史月份后自动重洗。

## 目标

1. 去掉金融渠道成本脚本中硬编码的月份常量。
2. 让脚本根据 `ods_bi_month_tag` 中实际存在的月份自动计算 `source_month` / `next_month` / `month_day_count`。
3. 保持现有分摊口径不变，只调整月份驱动方式。
4. 让历史月份补录后，下一次执行能够自动覆盖被影响的月份。
5. 删除类变更统一由编排层先清历史再触发对应月份重算，脚本本身不再依赖 `job_action`。

## 非目标

1. 不调整各成本项的分摊公式。
2. 不改动下游表结构和主键设计。
3. 不把批处理直接改成实时 CDC 任务。

## 方案

### 1. 月参数动态化

把原先固定写死的 `v_param` 改成从 `source_bi_month_tag` 聚合得到：

1. `source_month` 取 `statistics_time` 所在月的月初。
2. `next_month` 取下一个月月初。
3. `month_day_count` 取两个月初之间的天数。

### 2. 月份展开

把原先只生成单月日期的 `v_month_days` 改成按每个 `source_month` 展开整月日期。

### 3. 回刷方式

每次执行脚本时，扫描 `ods_bi_month_tag` 的全部有效数据，自动产出所有出现过的月份。
这样当财务在 7 月 8 日补录 6 月数据时，脚本下一次执行会自然重算 6 月对应的金融渠道成本。

### 4. 删除类变更

编排层在识别到软删除或硬删除后，先清理目标表中对应月份的历史数据，再调用脚本重跑对应月份。
脚本不再携带 `job_action` 开关，删除类场景只通过“先删后重跑”完成闭环，避免误灌新数据。

### 5. 硬删除对账

硬删除不依赖 `delete_time`，而是由编排层对原始表 `ods_bi_month_tag` 与目标表 `dwm_finance_channel_cost_p` 做月份维度的对比。
对比时优先按 `source_month` 汇总，再结合 `product_line`、`provider`、`source_tag`、`cost_type` 判断是否存在缺口或多灌。
一旦确认某个月份的指标在原始表侧消失，即视为该月受硬删除影响，编排层清理目标表对应月份历史数据后结束，不再插入新数据。

## 风险与约束

1. 如果 `ods_bi_month_tag` 里同时存在多个历史月份的数据，脚本会一次性覆盖多个 `source_month`，运行量会比单月大。
2. 现有幂等清理逻辑需要按 `source_month` 维度执行，否则可能残留旧数据。
3. 若部分脚本仍保留硬编码月份，整体一致性会被破坏，需要同步检查。
4. 硬删除无法仅靠当前 `ods_bi_month_tag` 表感知，必须依赖 CDC、审计日志或快照对比补足被删行信息。
5. 原始表与目标表的月份对账需要定义统一口径，否则容易出现“同月不同维度”误判。

## 验收标准

1. 脚本中不再出现固定的 `2026-05-01` / `2026-06-01` 月份常量。
2. `ods_bi_month_tag` 新增或修改历史月份数据后，脚本可自动覆盖对应月份。
3. 原有成本计算结果在单月场景下与改造前保持一致。
4. 删除类变更触发时，编排层先完成目标表历史清理，再触发脚本重跑对应月份，不再需要 `job_action`。
5. 硬删除场景可通过原始表与目标表的月份维度对账准确定位受影响月份。
